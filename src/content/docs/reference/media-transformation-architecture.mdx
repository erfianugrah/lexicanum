How [**video-resizer**](https://github.com/erfianugrah/video-resizer) handles media transformation on Cloudflare Workers.

Single endpoint for video resize, frame extraction, spritesheet generation, and audio extraction. Configuration lives in KV so you don't need to redeploy when changing settings. Two-layer cache (CDN edge + KV) with param validation before hitting `cdn-cgi/media` for transformations. Videos exceeding the cdn-cgi size limit are handled by a Cloudflare Container running ffmpeg.

---

## Overview

Five transformation strategies:

- **Video**: Resize, compress, adjust quality/playback via `cdn-cgi/media`
- **Frame**: Extract stills at timestamps (jpg/png)
- **Spritesheet**: Grid previews for scrubbing UIs
- **Audio**: Extract m4a audio tracks
- **Container (FFmpeg)**: Fallback for oversized videos that exceed `cdn-cgi/media` limits

Config-driven mode enablement, strong validation, mode-specific strategies. Each mode handles its own param prep and validation (strategy pattern).

## Core Concepts

### Request lifecycle

| Step         | Action                   | Details                                                     |
| ------------ | ------------------------ | ----------------------------------------------------------- |
| 1. Parse     | Normalize query params   | Config, derivatives, IMQuery, aliases, mode inference       |
| 2. Cache     | KV cache check           | Dimension-normalized keys (derivatives resolved to w/h)     |
| 3. Select    | Choose strategy          | Video / Frame / Spritesheet / Audio                         |
| 4. Build     | Create cdn-cgi URL       | Cache version, origins, path resolution                     |
| 5. Transform | Fetch via cdn-cgi        | Or container fallback on 9402 error                         |
| 6. Store     | KV streaming chunk store | 5 MiB chunks, pull-based reads, FixedLengthStream responses |
| 7. Shape     | Format response          | Headers, filename, range request support                    |

### Design patterns

| Pattern                     | Purpose                                  | Benefit                                       |
| --------------------------- | ---------------------------------------- | --------------------------------------------- |
| **Strategy per mode**       | Each mode = own validation + params      | Add modes without touching handler            |
| **Reactive error handling** | Container fallback on cdn-cgi 9402 error | No pre-flight HEAD, works for R2              |
| **Config + schema**         | Zod validation + runtime defaults        | New modes work even if config lags            |
| **Translation layer**       | Map Akamai/short params to internal      | Support multiple param conventions            |
| **Cache key normalization** | Derivatives resolved to dimensions       | `?imwidth=1280` and `?width=1280` share cache |

## Container FFmpeg Fallback

### Why

`cdn-cgi/media` rejects inputs exceeding its size limit (256 MiB for this account, 100 MB default) with error 9402 "Origin Too Large". Previously these were served untransformed. Now a Cloudflare Container running ffmpeg handles them.

### How it works

The container path is **reactive** -- it lives in the error handler, not in a pre-check:

1. `cdn-cgi/media` attempts the transformation normally
2. For oversized inputs, cdn-cgi returns HTTP 400 with CF error code 9402
3. The error handler detects 9402 and checks if the container is enabled
4. For R2 sources, constructs a `__r2src` URL the container can fetch
5. Sends a POST to the container's `/transform` endpoint with resize params
6. Container downloads source, runs ffmpeg, streams output back
7. Worker tees the response: one leg to client, one to KV streaming storage
8. Container response returns directly (bypasses error handler finalization to avoid `Cache-Control: no-store` and fallback headers)
9. Subsequent requests hit KV cache

### Container spec

- **Image**: Alpine + Node 22 + ffmpeg (`container/Dockerfile`)
- **Instance type**: `standard-4` (4 vCPU, 12 GiB RAM, 20 GB disk)
- **Sleep**: 5 minutes idle timeout, scales to zero
- **Max instances**: 5 concurrent
- **Endpoint**: `POST /transform` with JSON body (sourceUrl, width, height, quality, fit, etc.)
- **Output**: Streams MP4 with `Content-Length`, `-movflags +faststart`, stereo AAC audio

### ffmpeg command

```bash
ffmpeg -i input.mp4 \
  -vf "scale=W:H:force_original_aspect_ratio=decrease" \
  -c:v libx264 -preset medium -crf {18|23|28} \
  -c:a aac -b:a 128k -ac 2 \
  -movflags +faststart \
  output.mp4
```

Quality mapping: low=CRF 28, medium=CRF 23, high=CRF 18. Audio is always downmixed to stereo for web playback.

## Configuration

### Parameters

| Param      | Values                                         | Range/Rules                                              |
| ---------- | ---------------------------------------------- | -------------------------------------------------------- |
| `mode`     | `video` \| `frame` \| `spritesheet` \| `audio` | Auto-set to `audio` if `format=m4a`                      |
| `format`   | `jpg` \| `png` \| `m4a`                        | Frame: jpg/png, Audio: m4a, Video: n/a, Spritesheet: n/a |
| `time`     | `0s` - `10m`                                   | Default: `0s` (start position)                           |
| `duration` | `1s` - `300s`                                  | Omit = full length (up to platform limits)               |
| `width`    | `10` - `2000`                                  | Required for spritesheet                                 |
| `height`   | `10` - `2000`                                  | Required for spritesheet                                 |
| `fit`      | `contain` \| `scale-down` \| `cover`           | Video/Frame/Spritesheet only                             |

:::note[Audio mode restrictions]
Audio disallows: `width`, `height`, `fit`, `quality`, `compression`, playback params
:::

### Cache model

**KV keys** are normalized to dimensions (derivatives are resolved):

```
video:sample.mp4:w=1280:h=720:c=auto
frame:clip.mp4:t=5s:w=640:h=360:f=png
audio:rocky.mp4:duration=120s:t=30s:f=m4a
```

Both `?width=1280&height=720` and `?imwidth=1280` (resolved via the `tablet` derivative to 1280x720) produce the same cache key. One transformation serves both param conventions.

**Chunked storage**: Files over 20 MiB are stored as 5 MiB chunks with a JSON manifest at the base key. The streaming chunk processor reads the response body incrementally -- memory usage is bounded at ~10 MiB regardless of file size.

**cdn-cgi URL**: Includes all transform params + `version` for cache busting.

**Headers**: Auto-corrected `Content-Type` per mode, optional `Content-Disposition` filename.

### KV chunked read/write

**Storage** (`streamStorage.ts`): `StreamingChunkProcessor` accumulates incoming stream data into a 5 MiB buffer, copies it via `.slice()`, and uploads each chunk through a concurrency-limited queue (max 2 concurrent KV writes). A manifest JSON with chunk sizes is stored at the base key.

**Retrieval** (`getVideo.ts`):

- **Full content (200)**: Pull-based `ReadableStream` with `pull()` that fetches one KV chunk per call. Piped through `FixedLengthStream` so Cloudflare preserves `Content-Length`.
- **Range requests (206)**: Pre-calculates which chunks overlap the byte range, fetches only those via pull-based stream, slices at boundaries. Also uses `FixedLengthStream`. Browsers need `Content-Length` on 206 responses for video seeking.
- **Full-file ranges** (`Range: bytes=0-`): Falls through to the full content path (200) to avoid the browser getting stuck on a single 206 response covering the entire file.

### R2 source URL resolution (`__r2src`)

`cdn-cgi/media` needs a valid HTTP URL as the source video -- it sends HEAD/GET probes to fetch the content for transformation. R2 bucket bindings aren't HTTP-addressable, so R2 sources use a self-referencing URL pattern with a `__r2src` query parameter.

**The problem:** R2 bindings produce `r2:path` references, not HTTP URLs. Passing these to `cdn-cgi/media` results in 9402 errors because cdn-cgi can't resolve them.

**The solution:** When building the source URL for an R2 origin, the worker generates `https://cdn.erfi.dev/{path}?__r2src={bucketBinding}`. When cdn-cgi sends its HEAD/GET probe back to this URL, the worker intercepts requests with `__r2src` early in `index.ts` and serves raw R2 content directly via the bucket binding -- bypassing the transformation pipeline entirely.

The same `__r2src` pattern is used by the FFmpeg container to fetch R2 sources -- the container calls this URL and the Worker serves the R2 object.

```d2
direction: right

Client -> Worker: /videos/rocky.mp4?width=640
Worker -> CdnCgi: /cdn-cgi/media/width=640/https://cdn.erfi.dev/rocky.mp4?__r2src=VIDEOS_BUCKET
CdnCgi -> Worker2: HEAD rocky.mp4?__r2src=VIDEOS_BUCKET
Worker2 -> R2: r2Bucket.head("rocky.mp4")
R2 -> Worker2: metadata
Worker2 -> CdnCgi: 200 OK + headers
CdnCgi -> Worker3: GET rocky.mp4?__r2src=VIDEOS_BUCKET
Worker3 -> R2b: r2Bucket.get("rocky.mp4")
R2b -> Worker3: stream
Worker3 -> CdnCgi: 200 OK + body
CdnCgi -> Client: transformed video

Client: Client
Worker: Worker (build cdn-cgi URL)
CdnCgi: cdn-cgi/media
Worker2: Worker (__r2src intercept)
Worker3: Worker (__r2src intercept)
R2: R2 Bucket
R2b: R2 Bucket
```

### Cache versioning and cdn-cgi cache busting

`cdn-cgi/media` has its own internal cache that can't be purged directly. Cache versioning solves this by appending a `v=N` query parameter to the source URL inside the cdn-cgi path.

When you purge KV cache entries, the next request sees a KV miss and increments the version (e.g. `v=1` to `v=2`). This changes the cdn-cgi URL, forcing it to treat the request as new and re-fetch + re-transform the source.

```
# v=1 (cached in cdn-cgi)
/cdn-cgi/media/width=640/https://cdn.erfi.dev/rocky.mp4?__r2src=VIDEOS_BUCKET&v=1

# After KV purge -> v=2 (cdn-cgi cache miss, fresh transform)
/cdn-cgi/media/width=640/https://cdn.erfi.dev/rocky.mp4?__r2src=VIDEOS_BUCKET&v=2
```

Versions are stored in the `VIDEO_CACHE_KEY_VERSIONS` KV namespace. Under normal operation, the version stays stable. It only increments when a KV cache miss occurs (i.e. after a purge).
